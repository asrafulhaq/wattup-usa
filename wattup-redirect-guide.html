URL Redirect Setup Guide

Fixing 404 errors after the Webflow to Next.js migration

After the migration from Webflow to Next.js, a number of page URLs changed. Any links pointing to old Webflow URLs from Google, external platforms, or other sites now return 404 errors. This guide covers how to set up 301 permanent redirects so those old URLs forward correctly to the right pages, and Google updates its index accordingly.
Step 01

Audit All Broken URLs

Before writing any redirects, collect every URL that is currently returning a 404. There are three ways to do this:

Google Search Console (Recommended)
  1. Open search.google.com/search-console
  2. Go to Indexing → Pages
  3. Filter by Not found (404)
  4. Export the full list
Vercel Logs
  1. Open the WattupUSA project on vercel.com
  2. Go to the Logs tab
  3. Filter by status code 404
  4. Note all failing request paths
Old Webflow Sitemap
  1. Visit the old sitemap at https://your-webflow-domain.com/sitemap.xml
  2. Or look it up on the Wayback Machine: web.archive.org
  3. Use that as the source of truth for old URLs
Step 02

Create the Redirects File

Create a file called redirects.json in the root of the Next.js project. Each entry maps an old Webflow path to the new Next.js path.

// redirects.json
[
  { "source": "/about-us",      "destination": "/about" },
  { "source": "/our-work",     "destination": "/portfolio" },
  { "source": "/blog/:slug",   "destination": "/articles/:slug" },
  { "source": "/contact-us",  "destination": "/contact" }
]
Note source is the old Webflow path. destination is the new Next.js path. Use :slug syntax for dynamic segments where the URL pattern is the same but the folder changed.
Step 03

Update next.config.js

Load the redirects file and wire it into the Next.js config. Every redirect is set to permanent: true which sends a 301 status, telling Google to transfer SEO authority to the new URL.

// next.config.js

const redirectsList = require('./redirects.json');

const nextConfig = {
  async redirects() {
    return redirectsList.map(({ source, destination }) => ({
      source,
      destination,
      permanent: true,
    }));
  },

  // ... rest of your config
};

module.exports = nextConfig;
Step 04

Test Locally

Start the dev server and verify each redirect works before deploying.

npm run dev

Then test with curl or open the old URL in the browser. It should forward to the new page:

# Should return HTTP 301 and redirect to the new URL
curl -I http://localhost:3000/old-webflow-slug

You can also install the Redirect Path Chrome extension to see the redirect chain visually in the browser.

Step 05

Deploy to Vercel

Commit both files and push. Vercel picks up next.config.js redirects automatically and serves them at the CDN edge, so no additional configuration is needed.

git add redirects.json next.config.js
git commit -m "fix: add 301 redirects for Webflow to Next.js URL changes"
git push
Step 06

Post-Deploy Checklist

After the deploy goes live, complete these steps to restore SEO fully.

  • Test a few old URLs live and confirm they redirect to the correct new pages
  • Submit the new sitemap in Google Search Console under Sitemaps: https://wattupusa.com/sitemap.xml
  • Use the URL Inspection tool in Search Console to request re-indexing for the most important pages
  • Monitor the Coverage report over the next 1 to 2 weeks. The 404 count should drop as Google recrawls.

Important

Things to Watch Out For

Do not mix redirect configs Do not add redirects to both next.config.js and vercel.json. This causes conflicts. Stick to next.config.js only.
Always use permanent: true Setting permanent: false sends a 302 (temporary) redirect. Google will not update its index and SEO authority will not transfer. Always use permanent: true for this migration.
Paths are case-sensitive Make sure the source path in redirects.json matches the old URL exactly, including casing.
500+ redirects If the number of redirects grows beyond 500, consider switching to Vercel Edge Config with Middleware for better performance. For most sites, the next.config.js approach handles this volume well.
